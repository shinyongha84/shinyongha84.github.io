ㆍSpring Boot
JWT Token 을 생성하여 유저별로 발급을 진행하고, Client 의 요청에 담겨있는 JWT Token 을 검사하여 유효한 요청인지를 확인하는 Interceptor 를 개발하였습니다.
만약 유효하지 않다면, 그에 맞는 Http Status Code 를 전송해 Client 에서 재발급을 유도할 수 있게끔 개발하였습니다.
JWT Token 의 생존 주기가 길지는 않으나 혹시 모를 탈취 위험에 대비해 Redis 를 이용하여 이용 가능한 Token 들에 대해 관리를 할 수 있게 개발하였습니다.
그리고 Spring Boot 의 WebSocket 을 이용하여 결제 완료 여부를 전송하는 등 중요 로직에 대해 개발하였습니다.
그 외에는 Retrofit 을 이용해 Main Server 로 API 를 보내는 로직을 개발하였습니다.

Project Issue

ㆍ새로고침시 데이터를 잃어버리는 현상
결제모듈을 개발하는 도중 결제모듈에서 결제가 완료되면 Refresh Status Code 를 전달하여 데이터가 모두 초기화 되는 이슈가 있었습니다.
해당 증상을 막고자 Native Application 에 핵심 변수를 저장해놓고, Refresh 감지 및 결제 종료 값이 WebSocket 으로 전달된다면 Native 에서 꺼내와 변수를 다시 불러오는 방법으로 해결했습니다.

ㆍTypeScript 에서 외부 Library 참조
JavaScript 에서는 외부 Library 를 Import 하고 new 생성자를 통해 불러오면 됐지만, TypeScript 에서는 불가능 했습니다.
때문에 해당 문제를 해결하느라 여러 방법을 시도했었는데 'declare var ~~' 문을 통해 해결하였습니다.

ㆍPerformance
사용자의 기기마다 성능이 틀리기 때문에 저희는 속도를 저하 시키는 로직을 제거 했어야 했습니다.
속도 향상을 위해 forEach 문을 전부 제거하고, for 문으로 교체하고 Image 를 모두 최대한 가볍게 만드는 작업을 진행했습니다.

Project Review

ㆍ아쉬운 부분
컴포넌트 분리가 제대로 이루어지지 않았고, 외부 라이브러리를 끌어와 사용하는 작업이 많다보니 코드가 깔끔하지 않아 아쉬웠습니다.
서버가 2개로 분리되어 운영이 되고 있는데, 해당 부분이 사실 너무 과한 작업이 이루어지지 않았나 싶습니다.
이 당시 Subway KIOSK 작업을 할 때 처럼 Gradle 로 Build 파일을 하나로 합칠 수 있는 것을 알았다면, 서버가 1대만 들어가 관리 면에서 용이 했을 것 이라 생각합니다.
당시 방법을 몰라 헤매다가 결국 Front / Back 서버를 분리 시켰던 점이 아쉽습니다.

ㆍ전체적인 Review
JWT Token 을 개발하는데 꽤나 머리가 아팠던 것으로 기억합니다.
그럼에도 JWT Token 기능을 십분 활용하지 못하고, 일부만 활용된 것을 보면 아쉬울 뿐 입니다.
(JWT 내에 ID 만 담고 사용하고 있었는데, 패스워드도 BCrypt 처리 하여 JWT Payload 에 같이 담아 사용했다면 더 좋은 결과가 있지 않았을까 합니다.)
Native Application 을 만들고 앱스토어/구글플레이스토어 에 출시하는 것은 꽤나 어려운 작업이었던 것으로 기억에 남습니다.
안드로이드는 비교적 쉽게 출시를 진행했는데, 앱스토어에 출시를 하기 위해서 여러 인증서를 등록하고 컨펌받고 했던 것을 생각하면 좋은 경험인 것 같습니다.
해당 프로젝트에 TDD 도입이 되어 있다면 작업이 매끄럽게 흘러가지 않았을까, 생각도 되며 TDD 의 필요성이 계속 와닿고 있습니다.

결제 모듈을 사용하는 것은 상당히 힘들었습니다. 여러 난관이 있었는데, 잘 헤쳐 나가 마무리를 지었습니다.
(하지만 출시 2일을 앞두고 모듈이 바뀌면서 다시 개발 하고 iOS 심사 받느라 계속 모니터링 했던 것을 생각하면 잘 마무리가 된 것도 아니지 않나 싶습니다..)

처음으로 Android & iOS 를 불특정 사용자에게 배포해본 특별한 프로젝트입니다.
