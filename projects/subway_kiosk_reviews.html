My Work content

제가 맡은 업무는 Kotlin(Spring Boot) / Core Module 개발 / Build & Deploy Logic / Front-end(React.js) 개발 업무 입니다.

ㆍSpring Boot
데이터베이스와 연동하여 JPA/MyBatis를 사용해 필요에 따라 데이터베이스에서 값을 읽거나, 작성하는 서비스를 개발하였습니다.
Front ↔ Back 간 통신할 수 있는 API 를 개발해 주고, 상황에 따라 Core Module 을 실행시키는 작업을 개발하였습니다.
Spring Boot Scheduler Annotation 을 이용해 판매데이터를 자동으로 서버로 전송하는 작업도 개발중에 있습니다.
Front에서 Axios를 이용해 API를 호출하면 결제 모듈을 호출하여 결제를 진행시키고, 결제 여부를 Front에 내려주는 작업도 하고 있습니다.
결제를 진행 후 결제 데이터를 데이터베이스에 저장하고, 저장 되어 있는 결제 내역과 실제 결제를 Validation 하여 민감한 부분인 결제에 대해 안정성을 조금 더 늘리고자 노력했습니다.
Exception과 Exception Code를 많이 분리하여 조금 더 안정적인 시스템을 만들어보고자 하고 있습니다.
이 모든 API들은 Swagger로 관리되고 있습니다.

ㆍCore Module
타업체에서 전달해준 C++ DLL 파일을 JNA 로 불러와 결제모듈을 호출하는 모듈을 개발하였습니다.
Retrofit 을 이용해 타업체에서 전달해준 Interface 양식대로 API 를 호출하여 결과값을 Return 해주는 모듈을 개발하였습니다.
현재 Local Storage 에 있는 sha512 파일과 AWS S3에 접속하여 최신 sha512 파일을 비교하여 업데이트를 진행하는 로직을 개발하였습니다.
Core Module 들은 Facade 패턴을 이용하여 사용하기 때문에 Front 에서 복잡한 데이터를 보내지 않아도 되게 개발하였습니다.
또한 Core Module 은 모두 Project 를 분리시켜 코드관리에 신경썼으며, 해당 Module 들은 Maven Publish 를 통해 사내 Nexus 에 올라가게 하였습니다.

ㆍBuild & Deploy Logic
빌드 로직은 Gradle Kotlin DSL 을 이용해 개발하였습니다.
Gradle 에 신규 Task 를 생성하여 해당 Task 가 실행되면 React.js 를 Yarn Install → Yarn Build → File Copy 작업을 통해 Spring Boot 의 resources 폴더로 옮겨주고 있습니다.
위처럼 Build 를 진행하고 해당 파일을 bootJar Task 로 압축을 진행, 그리고 배포 로직을 실행하여 AWS S3 에 자동으로 업로드 되게 개발하였습니다.
이로써 Build & Deploy 자동화가 완료되었으며, Deploy 가 완료되면 Deploy 의 내용이 회사 협업 커뮤니케이션 툴에 메시지를 전송하여 유지보수에 용이하게 하였습니다.

ㆍFront-end(React.js)
REST Api를 이용해 Spring boot에서 데이터를 끌어와 Reducer에 저장시켜놓고 데이터를 불러와 화면 설계 및 컴포넌트 작성을 진행하였습니다.
Reducer에 있는 내용을 끌어올 때에는 Redux를 이용해 작업했습니다.
즉, 데이터 관리 및 컴포넌트 개발을 진행하였습니다.

Project Issue

ㆍPerformance
직원이 사용하는 기기가 아닌, 일반 고객이 직접 사용할 기기라 아무래도 속도면에서 많은 이슈가 있었습니다.
기기 사양이 좋지는 않아 Spring Boot Application 을 실행하는데 약 2분정도 소요가 되어 이슈가 되었습니다.
해당 부분은 최대한 Module 을 분리하여 불러오는 방식으로 해결하였으며 실제로 약 1분정도로 단축되었습니다.
그 외에도 Annotation 을 관리하여 없어도 되는 Annotation 을 제거 및 다른 방법이 있겠지만 아직 방법을 잘 몰라 계속 찾아보고 있습니다.
또한, 코드가 비대해져 불러오는데 오래 걸리는 부분들을 최소화하여 로직을 개선 하고 있으며 Front 에서는 Lazy Loading, 반복문 개선 등을 진행하여 속도를 개선해보고 있습니다.

Project Review

아직 완성되지 않은 프로젝트라 리뷰할 내용은 크게 없지만, 최대한 작성해보았습니다.

ㆍ아쉬운 부분
원래 해당 프로젝트에 TDD 를 모두 적용할 계획이었습니다.(Front & Back)
하지만 프로젝트 진행 일정이 촉박함에 따라 TDD 를 적용하지 못했습니다.
꼭 해보고 싶었던 부분이었는데 못하게 되어 정말 아쉽습니다.
다음에 프로젝트를 진행하게 되거나 혹은 진행중인 개인 프로젝트에 적용할 수 있다면 꼭 적용하고 싶습니다.

그리고 해당 프로젝트에 Clean architecture 를 지키며 개발 하려 하였으나 제대로 이루어지지 않은 것 같습니다.
Domain(Business Rules) 영역과 Infrastructure 영역은 분리 되었으나, Dependency Rule 이 제대로 지켜지지 않았고, Domain 영역에 너무 많은 Dependency 가 들어간게 원인이 아닐까 생각하고 있습니다.

ㆍ전체적인 Review
이번 프로젝트는 정말 Front 와 Back 이 나뉘었다는 것을 알 수 있었습니다.
원래는 제가 Front 와 Back 모두 작업을 진행 하여서 그런 것을 느낄 수 없었는데, 이제는 팀원들에게 Front 작업을 맡기고, 제가 맡은 업무에만 집중하니 참 좋았습니다.
제가 Core Module 들을 개발하고, 설명해준 후 팀원들에게 어떤식으로 데이터를 보내면 모두 처리하게 해놓겠다. 등 정말 효율적인 커뮤니케이션이 이루어졌습니다.
또한, 코드 리뷰를 통해 서로 부족했던 부분이나 몰랐던 부분에 대해 같이 의논하며 작업을 진행했습니다.
아직 끝나지 않은 프로젝트지만 이대로 쭉 완성한다면 기간 내에 좋은 결과물이 나올 것이라 기대하고 있습니다.
이전 프로젝트에서도 TodoList 를 관리하였는데, 이번 프로젝트에서는 TodoList 뿐 아니라 화면별 기능 정의서까지 작성하여 개발 효율을 높인 것 같습니다.
